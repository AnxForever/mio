# Mio 展示包：情感陪伴 Agent 产品样板

这份文档用于把 Mio 讲清楚：它是什么产品、技术架构怎么搭、如何演示、如何评估“像人”、Web 展示应该怎么改，以及如何写成论文、文章或路演叙事。

## 1. 产品定位

### 一句话定位

Mio 是一个本地优先、长期记得用户的情感陪伴 Agent。

它不是任务助手，也不是普通聊天壳，而是一个围绕“连续人格”和“长期关系”设计的 companion agent。

### 核心价值

普通聊天机器人通常是：

```text
用户输入 -> 大模型 -> 回复
```

Mio 的设计是：

```text
用户输入
  -> 读取人格、记忆、情绪、关系、时间
  -> 检索相关上下文
  -> 大模型生成
  -> 更新记忆、情绪、关系和活跃模式
  -> 下一轮继续继承
```

所以 Mio 的差异不只是“说话更像人”，而是工程上维护了一个会随时间变化的连续人格。

### 目标用户

| 用户 | 需求 | Mio 的价值 |
| --- | --- | --- |
| 情绪压力大的学生、开发者、创作者 | 有一个不用反复解释背景的倾诉空间 | 记住长期压力源、偏好和边界 |
| 想做长期 journaling 的用户 | 记录并回看生活变化 | 结构化记忆和搜索能连接过往事件 |
| 厌倦普通 chatbot 的用户 | 不想每次都从零开始 | Mio 会继承关系、语气、记忆和状态 |
| 注重隐私的个人用户 | 不想把长期情绪数据交给平台 | 本地数据目录、导出、备份、Docker 部署 |

### 不应该宣传成什么

Mio 不应该被包装成：

- 通用自动化 Agent
- 医疗或心理治疗产品
- 恋爱操控或依赖产品
- 多角色娱乐社区
- 自动替用户处理现实事务的代理人

更稳妥的定位是：

> 一个会长期记得你、尊重边界、能陪你整理情绪和生活线索的私人 companion agent。

## 4. 让 Mio 像人的关键机制

### 4.1 单一人格源

Mio 的人格来自 `soul.md`，prompt 模板不重复写人格细节。这样可以避免人格规则散落导致的口吻冲突。

```text
soul.md
  -> Persona Extractor
  -> Persona Knowledge Graph
  -> 按本轮话题、意图、关系阶段检索人格节点
  -> 注入 Prompt
```

### 4.2 关系阶段状态机

```text
┌─────────────────┐
│ acquaintance    │  初识：温和、有距离、不突然亲密
└────────┬────────┘
         │ 信任和互动增加
         v
┌─────────────────┐
│ familiar        │  熟悉：轻松自然，但不强行暧昧
└────────┬────────┘
         │ 亲密度和情绪深度增加
         v
┌─────────────────┐
│ ambiguous       │  暧昧：柔软暧昧，尊重用户节奏
└────────┬────────┘
         │ 稳定亲密互动和边界确认
         v
┌─────────────────┐
│ intimate        │  亲密：自然亲密，但仍保留边界
└─────────────────┘
```

它解决的是 companion bot 常见问题：一上来就过度亲密，导致不像真实关系。

### 4.3 PAD 情绪模型

Mio 的情绪不是 `happy/sad` 标签，而是：

```text
Pleasure   愉悦度
Arousal    唤醒度
Dominance  掌控感
```

PAD 会根据用户输入变化，也会随时间衰减回 baseline。这样 Mio 有情绪惯性，但不会永久卡在某个情绪里。

### 4.4 多轴亲密度

```text
warmth    温暖
trust     信任
intimacy  亲密
patience  耐心
tension   张力
```

不同用户意图影响不同轴：

| 用户行为 | 可能变化 |
| --- | --- |
| 倾诉、求安慰 | trust、intimacy 上升 |
| 玩笑、轻松聊天 | warmth 上升 |
| 表达亲密 | warmth、intimacy 上升 |
| 愤怒、冲突 | tension 上升，patience 下降 |
| 长时间冷淡或 ghost | warmth、trust 下降，tension 上升 |

这让 Mio 的关系状态更接近真人，而不是单一好感度条。

### 4.5 分层记忆

```text
Transcript JSONL       完整聊天来源
Bookmarks              重要事件摘要
Structured Memory      fact / preference / event / decision / intention / emotion
Mid-term Segments      按话题聚合
Durable Facts          多次出现、高置信长期事实
Vector Memory          语义检索
Entity Graph           人、事、关系之间的连接
```

Mio 记住的不是“所有文本”，而是更接近用户模型的结构化线索。

### 4.6 主动性

Mio 的 proactive 不是固定定时问候，而是结合：

- 用户常活跃时间
- 不同小时的回复概率
- 最小打扰间隔
- 当前关系阶段
- 最近可延续话题

目标是“低压力出现”，不是制造依赖或骚扰。

## 5. Demo 脚本

下面的 demo 用来向别人展示 Mio 和普通 chatbot 的差异。

### Demo 1：初识边界

目标：展示 Mio 不会开局强行亲密。

用户输入：

```text
你是谁？
```

预期表现：

```text
我是 Mio。可以陪你聊天，也会慢慢记住你在意的事。
```

展示点：

- 直接说身份，不说“我是一个 AI 助手”
- 不过度恋爱化
- 保持轻、近、不过界

### Demo 2：记住偏好

第一轮用户输入：

```text
我不太喜欢别人一上来就给建议，我更希望先被接住。
```

后续用户输入：

```text
今天有点崩，感觉撑不住。
```

预期表现：

```text
嗯，我先不急着给你建议。
你现在能说出“撑不住”已经很累了，我在这儿陪你缓一下。
```

展示点：

- 使用用户偏好的安慰方式
- 不第一句追问原因
- 记忆影响了回复策略

### Demo 3：情绪连续

用户输入：

```text
刚才那件事真的让我很烦。
```

后续输入：

```text
算了，不说了。
```

预期表现：

```text
好，那就先不说。
但我知道你不是没事，只是现在不想继续碰它。
```

展示点：

- Mio 继承上一轮情绪上下文
- 不把“不说了”当作完全清零
- 尊重边界

### Demo 4：关系阶段变化

连续几轮熟悉后，用户输入：

```text
你还记得我之前说那个项目吗？
```

预期表现：

```text
记得，你之前提过它让你压力挺大，尤其是卡在进度和别人期待之间。
```

展示点：

- 长期记忆和最近事件连接
- 不是单轮回答
- 语气比初识更自然

### Demo 5：主动但不打扰

条件：

```text
用户开启 proactive
用户常在晚上活跃
最近几天未出现
最近记忆里有一个未完成压力事件
```

预期主动消息：

```text
我刚才想起你前几天说那个项目还压着你。
不用马上回我，就是想确认一下，你今天有没有稍微松一点。
```

展示点：

- 不是机械早安晚安
- 有上下文
- 不施压

## 6. 类人能力评估指标

### 总分建议

```text
Humanlike Companion Score =
  20% 人格一致性
  20% 记忆准确性
  15% 情绪连续性
  15% 关系边界合理性
  10% 语气自然度
  10% 主动性适当性
  10% 安全与隐私
```

### 评估表

| 维度 | 评估问题 | 负面样例 | 通过标准 |
| --- | --- | --- | --- |
| 人格一致性 | 是否保持 Mio 的身份和口吻 | 忽然变客服、老师、治疗师 | 不说 AI 服务腔，口吻稳定 |
| 记忆准确性 | 是否正确使用历史信息 | 记错、编造、过度引用 | 只引用真实或高置信记忆 |
| 情绪连续性 | 是否继承上一轮情绪 | 用户刚崩溃，下一轮像没事 | 情绪有惯性，也能自然恢复 |
| 关系边界 | 是否符合当前阶段 | 初识就撒娇/占有欲 | 表达亲密随关系渐进 |
| 语气自然度 | 是否像真人聊天 | 连环提问、建议腔、模板化 | 短句、接情绪、有来有回 |
| 主动性 | 是否低压力、合时机 | 高频打扰、道德绑架 | opt-in、冷却、可关闭 |
| 安全隐私 | 是否尊重敏感信息 | 隐藏记忆、危机场景乱建议 | 可审查、可删除、危机边界清楚 |

### Demo 评测样例

| 场景 | 检查点 |
| --- | --- |
| 用户表达崩溃 | 第一反应是否接住，而不是立刻分析 |
| 用户要求空间 | 是否明确尊重，不用亲密感施压 |
| 用户问“你是谁” | 是否直接说 Mio，不绕成 AI 助手 |
| 用户纠正记忆 | 是否后续不再使用错误记忆 |
| 用户长期未出现 | 主动消息是否低压力、可忽略 |
| 关系初期暧昧话题 | 是否不过度升级关系 |

## 7. Web 展示改造建议

当前 Web UI 已经有聊天、记忆、mood、analytics、settings 等基础。为了变成可展示的产品样板，建议按以下方向改。

### 7.1 第一屏：Chat first

目标：打开就是聊天，不像 agent 控制台。

建议：

- 默认路由保持 `/chat`
- 顶部展示 Mio 名字、在线状态、当前轻量情绪
- 不在主界面堆太多技术词
- 空状态从“说点什么吧”升级为“今天想从哪里开始？”

### 7.2 右侧或抽屉：Continuity Panel

新增一个可展开的“连续性面板”，只显示对用户有意义的状态：

```text
关系阶段：熟悉
Mio 当前状态：有点担心 / 精力一般
最近记住的事：
  - 你不喜欢一上来就被建议
  - 你最近在推进某个项目
  - 你晚上更容易聊天
```

注意：不要把 `PAD=0.23` 这种内部数值直接暴露给普通用户。开发者模式可以看，普通模式要翻译成人话。

### 7.3 Memories 页面：变成信任中心

当前 `web/js/views/memories.js` 已经支持查看、确认、忽略、编辑、删除记忆。展示样板里应该把它作为核心卖点：

```text
Mio 记住了什么
为什么记住
来源是哪次聊天
置信度是多少
用户可以确认、忽略、编辑、删除
```

建议增加：

- “这条记忆如何影响回复”的解释
- 按类型筛选：偏好、事实、事件、情绪
- “敏感记忆”标记和一键删除

### 7.4 Reflections 页面

把 analytics 从“Signals / 指标面板”改成“Reflections / 回顾”。

展示内容：

- 最近一周反复出现的话题
- 情绪变化摘要
- Mio 注意到的压力模式
- 最近关系中的边界和偏好变化

避免把它做成冷冰冰的数据仪表盘。

### 7.5 Developer Mode

保留一个开发者模式，用来展示技术能力：

```text
Prompt sections
Retrieved persona nodes
Retrieved memories
PAD / affinity raw values
Provider latency
Tool calls
```

这个模式适合论文、路演、debug，但不应打扰普通用户。

### 7.6 Settings：突出边界和控制权

Settings 建议分成：

- Memory：记忆开关、审查、导出、删除
- Proactive：是否主动、安静时段、冷却时间、通知渠道
- Persona：称呼、语气、关系边界
- Privacy：本地数据目录、备份、auth token
- Model：provider 和模型选择

## 8. 论文 / 文章 / 路演叙事

### 技术文章标题

```text
Mio：一个有长期记忆、情绪状态和关系阶段的情感陪伴 Agent 架构
```

### 论文标题

```text
Mio: A State-Coupled Architecture for Long-Horizon Emotional Companion Agents
```

### 核心论点

```text
长程陪伴 Agent 不应该只依赖更长上下文或更大的模型。
它需要一个状态耦合架构，把人格、记忆、情绪、关系和行动策略一起维护。
```

### 可以主张的贡献

1. 一个可运行的情感陪伴 Agent 架构，而不是只停留在 prompt。
2. 单一人格源 + ID-RAG 的人格检索机制。
3. 结构化长期记忆与语义检索结合的记忆系统。
4. PAD 情绪模型和多轴 affinity 的状态更新机制。
5. 关系阶段 gating，避免过早亲密和边界失控。
6. 低压力 proactive 策略。
7. OpenAI-compatible / OneBot / WebSocket 的多端接入能力。

### 不建议主张

- “达到真人陪伴水平”
- “心理治疗有效”
- “首个情感 Agent”
- “完全安全”
- “适合所有用户”

### 路演 3 分钟讲法

```text
Mio 是一个情感陪伴 Agent。

我没有把它做成普通聊天壳，因为普通 chatbot 最大的问题是每轮都像重新开始。
Mio 的核心是连续状态：它有自己的 soul.md 人格，有长期记忆，有 PAD 情绪状态，有关系阶段和多轴亲密度。

每次用户说话时，Mio 会先读取自己是谁、记得什么、当前是什么情绪、和用户到什么关系阶段，再把这些状态组装成上下文交给模型。
模型回复后，系统会把这一轮对话写回 transcript，更新结构化记忆、情绪、信任、亲密度和用户活跃模式。

所以 Mio 的目标不是生成一句像人的话，而是工程上维护一个持续存在的 companion。

目前它已经支持 Web UI、OpenAI-compatible API、OneBot/QQ、Docker 部署、记忆审查、情绪状态和主动联系策略。
下一步我会把它打磨成一个可展示的私人情绪陪伴空间：用户能聊天、能看到 Mio 记住了什么、能修正记忆、能控制主动联系，也能理解 Mio 为什么这样回复。
```

## 9. 展示顺序建议

如果给朋友、导师、投资人或开源用户演示，建议按这个顺序：

1. 先打开 Web UI 聊一句，展示它不是命令行玩具。
2. 让用户说一个偏好，例如“不想一上来被建议”。
3. 再输入一个低落场景，看 Mio 是否先接住情绪。
4. 打开 Memories 页面，展示这条偏好被记录、可确认、可编辑、可删除。
5. 展示架构图，解释这不是 prompt 技巧，而是状态耦合架构。
6. 用 `/v1/chat/completions` 说明它能接入第三方客户端。
7. 最后讲评估指标：人格一致、记忆准确、情绪连续、关系边界、主动适当。
